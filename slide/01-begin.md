# AIエージェントとは

- 目標達成のために外部ツールと文脈で反復実行する仕組み
- ユーザーを補助し、繰り返し行うべきコンピュータ関連のタスクをユーザーに代わって行う
- 得手不得手: パターン抽出◎ 長期記憶/曖昧要件△

<!--
- 「自走する開発者」ではなく“制御可能な実行エンジン”。
-->

---

# アーキテクチャの視点

- System方針 → Task指示 → コンテキスト供給 → 実行 → 評価
- 重要: 最小十分の入力/検査可能な出力/差分適用
- メトリクス: 再現度・差分量・コミット数

Note:

- 各レイヤの責務を具体例で示す。

---

# AIエディタ・ツールの使い分け

- Cursor: 編集/リファクタ/差分適用が強み
- Claude Code: 仕様化/要約/長文合成が強み
- CLI=再現性/ログ、GUI=探索速度

<!--
- GUIで探索→CLIで確定、の二段運用を推奨。
-->

---

# Cursor のポイント

- 機能: 補完/エージェント編集/マルチファイル差分
- コツ: 目的/制約/影響範囲を先に固定、小さなコミット
- 例: AGENTS.md の特定見出しのみ編集 → Prettier
- 補足: コンテキストは最大600k（約45万単語）

Note:

- 「範囲指定→意図→非目標→成功条件」で依頼する。

---

# Claude Code のポイント

- 強み: 文脈保持と説明/要件擦り合わせ
- 用途: 仕様ドラフト、テキスト→チェックリスト化
- 注意: 章立てと合否基準を先に決める

Note:

- 冗長指示は逸脱の原因。構造で制御する。

---

# OSSエージェント: [Serena](https://github.com/oraios/serena)

- **ローカルで動作するMCPサーバー**
  - OSSのコーディングエージェントツールキット
  - APIキー不要でコストを気にせず利用可能
- **言語サーバー(LSP)との連携**
  - コードをテキストではなく「意味」で理解
  - シンボル単位の検索、参照追跡、リファクタリングが可能
  - 例: `find_referencing_symbols`, `replace_symbol_body`
- **高い拡張性と透明性**
  - 独自のツールを追加して機能を拡張できる
  - 特定のIDEやLLMに依存しない柔軟な設計
  - 内部動作を理解・改変できる

Note:

- 自前の開発環境にAIを深く統合したい場合に最適。
- セットアップは必要だが、コストと拡張性が魅力。

---

# 外部知識注入: [Context7](https://github.com/upstash/context7)

- **最新のライブラリドキュメントを提供するMCPサーバー**
  - AIの知識の古さを補い、正確なコード生成を支援
- **プロンプトからドキュメントを自動取得**
  - `get-library-docs` ツールでライブラリの公式ドキュメントを検索
  - 使い方、API仕様、設定方法などをコンテキストに注入
- **得意なこと**
  - 「Xライブラリの最新版で認証機能を実装して」
  - 「Yのセットアップ手順を教えて」

Note:

- Serenaが「コード構造の理解」なら、Context7は「外部知識の検索」。
- 組み合わせることで、より高度な開発支援が期待できる。

---

# ブラウザ操作: [Playwright MCP](https://github.com/microsoft/playwright-mcp)

- **AIにブラウザ操作能力を付与するMCPサーバー**
  - E2Eテストフレームワーク Playwright をAIから利用
- **ブラウザ上のあらゆる操作をツール化**
  - `browser_navigate`: ページ遷移
  - `browser_snapshot`: ページ構造の認識（AIの目）
  - `browser_click`, `browser_type`: 要素の操作
- **得意なこと**
  - 「サイトにログインして情報を要約して」
  - 「フォームに入力するテストコードを生成して」

Note:

- コード(Serena), 知識(Context7), Web(Playwright)をAIが扱える。
- WebサイトのUIテストや情報収集を自然言語で指示可能に。

# 速度と品質の原則

- 小さく切る: 1出力=1意図
- 前提固定: 成功条件/禁止事項/非目標
- 検証優先: 整形/リンタ/プレビューで即確認

Note:

- マントラを口頭で復唱し定着させる。

---

# コンテキスト設計 1/3（情報源）

- 仕様/規約/資産/制約を棚卸し
- 必須/参考/禁止/未確定でラベル分け
- 不要な長文はリンク化

Note:

- 未確定を前提にしない。仮置きは明記。

---

# コンテキスト設計 2/3（提示）

- 構造: System/Task/Evaluation
- 粒度: 抜粋+行番号/見出しで指示
- 例: 「言語ポリシーに箇条書きを1点追加」

Note:

- 出力フォーマットを先に固定して自由度を絞る。

---

# コンテキスト設計 3/3（リポジトリ）

- AGENTS.md に規約と言語方針を集約
- 命名: lower-kebab-case.md / 1H1 原則
- 構造: slide/=発表, guide/=配布, root=基礎

Note:

- 構造＝意思表示。他者とAIが迷わない入口を作る。

# MCP 概要と導入

- 目的: 取得しづらい外部データを安全に注入
- ガード: 最小権限/読み取り優先/アクセスログ
- 流れ: 設定→検証→運用（撤退条件を文書化）

Note:

- セキュリティ不安を払拭。まず読み取り専用で開始。

---

# デザイン→コードの流れ

- 参照軸: レイアウト/スケール/タイポ/カラー/余白
- 手順: 仕様抽出→トークン→コンポーネント→ページ
- 検証: 目視+ルーラー+スクショ差分（概念）

Note:

- まずトークン化→最後にページ組み。順番が肝。

---

# デモ1（素のアプローチ）

- 入力: LP要件（セクション/コピー/配色）
- 生成: 静的 HTML/CSS 骨組み
- 検証: 構造/階層/余白/タイポの差分洗い出し

Note:

- 敢えて曖昧さを残し、崩れ方を観察する教材にする。

---

# デモ1の学び

- 曖昧な入力→再現度低下
- 先に成功条件/禁止事項/計測方法を決める
- 次: コンテキスト縮減と指示の正規化

Note:

- 「曖昧さリスト」を作り、次の依頼に反映。

---

# デモ2（MCP活用）

- 準備: トークン/グリッドをMCPで参照
- 指示: トークン→変数→基礎コンポーネント→ページ
- 検証: トークン一致率/API一貫性/差分量

Note:

- 実行前に合否基準を音読→合意→実行。逸脱は即中断。

---

# 結果比較と再現手順

- 指標: 再現度/コミット数/修正差分
- 観察: MCPで初期ブレ縮小、修正が局所化
- 再現: 指示テンプレ/チェックリスト/ロールバック手順

Note:

- 体感ではなく差分で語る。図示/スクショを用意。
